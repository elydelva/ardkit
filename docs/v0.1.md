# ADRKit — V0.1 Scope

> Deliverable tonight. Everything not in this document is v0.2.

---

## Goal of v0.1

Prove one thing: does an agent with `adrkit context` in its system prompt behave better than one without?

To answer that, exactly four commands need to work on a real realm. No TUI. No sync. No full linter. Just the minimum to use ADRKit on ADRKit itself tonight.

---

## What ships tonight

### The four core commands

```bash
adrkit init
# Creates .adrconfig with default values
# Creates .adrkit/ and .adrkit/.state
# Creates .adrkit/templates/ with base templates
# Success message + suggested next step

adrkit new
# Interactive: prompts for title, type, phase
# With flags: --title --type --phase --tags (non-interactive, agent-friendly)
# Creates ADR-XXXX-slug/ with ADR-XXXX.md scaffolded from template
# Increments counter in .adrkit/.state
# Automatic git add

adrkit context
# JSON dump of full realm: ADRs, tasks, statuses, active rules
# --adr ADR-0001 to scope
# --active to filter on in-progress only
# Designed to be piped into an agent: adrkit context --active | pbcopy

adrkit next
# Returns the next task to work on
# Simplified v0.1 algorithm: phase first → oldest ADR → oldest task
# Gates are respected (blocked task = skipped)
# --json for machine-readable output
```

### Basic task commands (minimum viable)

```bash
adrkit task add ADR-0001 --title "..." --type implementation
adrkit task complete TASK-0001
adrkit task start TASK-0001
```

### That's it.

## What is explicitly out of scope for v0.1

| Feature | Why deferred |
|---|---|
| TUI (`adrkit tui`) | Ink setup + components = minimum one day |
| Linear/GitHub sync | Depends on external tools, not testable in isolation |
| Git hooks (`adrkit hooks install`) | Useful but not blocking to validate the concept |
| Full linter (L001–L051) | Basic validation is sufficient in v0.1 |
| `.state` merge driver | Unnecessary in solo use |
| `adrkit history` | Traces are created, not yet queryable |
| `adrkit graph` | Visualization = comfort, not necessity |
| `adrkit lint` | Replaced by lightweight inline validation |
| `adrkit rules ack` | The rules system exists in files, not yet executed |
| Sync adapters | The entire `@adrkit/sync` package |

## v0.1 Architecture — the honest minimum

The monorepo is created in its final shape. But only three packages are active.

```
adrkit/
├── package.json              # workspaces configured for all packages
├── bunfig.toml
├── tsconfig.base.json
└── packages/
    ├── core/                 # @adrkit/core — active
    ├── fs/                   # @adrkit/fs — active
    ├── git/                  # @adrkit/git — active (staging only)
    ├── lint/                 # @adrkit/lint — stub, not implemented
    ├── tui/                  # @adrkit/tui — stub, not implemented
    ├── sync/                 # @adrkit/sync — stub, not implemented
    └── adrkit/               # binary — active
```

The `lint`, `tui`, `sync` packages exist (folder + `package.json` + empty `src/index.ts`) so the monorepo doesn't need restructuring in v0.2. They export typed stubs that satisfy the compiler.

### What is implemented in each active package

**`@adrkit/core` v0.1**

```
entities/       ADR, Task, Trace (types + constructors)
value-objects/  ADRId, TaskId, Status (enums + validators)
services/       dag.service.ts (gates only, not the full DAG)
                state-machine.service.ts (valid transitions)
ports/          IRealmRepository, IGitAdapter (complete interfaces — for v0.2)
use-cases/      CreateADR, CreateTask, CompleteTask, GetNext, GetContext
```

Ports are fully defined even if v0.1 adapters are simplified. This investment protects the architecture for v0.2.

**`@adrkit/fs` v0.1**

```
parsers/        frontmatter.parser.ts (gray-matter + strict typing)
                adr.parser.ts, task.parser.ts
serializers/    adr.serializer.ts, task.serializer.ts
config/         adrconfig.reader.ts, state.manager.ts
realm.repository.ts   complete IRealmRepository implementation
```

No traces in v0.1: trace files are created (for format compatibility) but not queryable.

**`@adrkit/git` v0.1**

A single file: `staging.ts`. `git add` on modified `.adrkit/` files via `Bun.spawn`. That's all.

**`adrkit` v0.1**

```
commands/
  init.ts
  new.ts
  context.ts
  next.ts
  task/add.ts, start.ts, complete.ts
container.ts    simplified DI — direct instantiation, no framework
cli.ts          commander or clipanion
index.ts        entry point
```

No `--actor` flag in v0.1. No `--human-confirmed`. These flags are reserved but not enforced.

## The shortcuts taken and why they don't break anything

**No queryable traces.** `TRACE-XXXX.md` files are created on every mutation (so the format is correct from day one), but `adrkit history` doesn't exist yet. Traces exist in the filesystem; they'll be queryable in v0.2.

**Minimal inline validation.** No full linter. When `adrkit task complete` is called on a task with unresolved gates, the command blocks with a clear error message. That's the only check in v0.1. Rules L001–L051 arrive in v0.2.

**Direct DI in `container.ts`.** No sophisticated injection pattern. Adapters are instantiated directly. The interface contract (`IRealmRepository`) is respected — swapping adapters in v0.2 changes nothing else.

**Git staging only.** No pre-commit hook, no merge driver. `adrkit` runs `git add` on modified files. The commit stays manual. Hooks arrive when there's something to protect.

## Definition of "done" for v0.1

v0.1 is done when:

- [ ] `adrkit init` runs in an empty repo and creates the expected structure
- [ ] `adrkit new --title "Test" --type tech-choice` creates a valid ADR
- [ ] `adrkit task add ADR-0001 --title "First task"` creates a task
- [ ] `adrkit task complete TASK-0001` completes it and creates a trace file
- [ ] `adrkit context --active --json` returns parseable JSON
- [ ] `adrkit next` returns `TASK-0001` in the correct order
- [ ] ADRKit is used on itself (this project's ADRs live in `.adrkit/`)

The last point is the most important. If you can't use ADRKit to track ADRKit's own development, v0.1 is not done.

## What v0.1 prepares for v0.2

Because ports are properly defined and the monorepo is in place, v0.2 can plug in without restructuring:

- `@adrkit/lint` → implementation of rules L001–L051
- `@adrkit/git` → pre-commit, commit-msg, pre-push hooks
- `@adrkit/tui` → Ink, file watching, DAG display
- `adrkit history` → query on existing trace files
- `--actor`, `--human-confirmed` → flags already reserved in CLI

Nothing written tonight will be thrown away in v0.2. That's the criterion.
